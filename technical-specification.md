## 1. Что такое качественное техническое задание (ТЗ) и почему это важно

Техническое задание — это документ, который фиксирует требования к проекту и ожидаемому результату. Проще говоря, это мост между тем, что вы хотите получить, и тем, что в итоге сделает разработчик.

### 1.1. Что должно быть в качественном ТЗ

| Требование к ТЗ | Что это значит |
|---|---|
| **Полнота** | Описаны все функции, экраны, сценарии использования. |
| **Однозначность** | Нет размытых формулировок («яркий», «удобный», «быстрый»). |
| **Проверяемость** | Каждый пункт можно проверить (работает/не работает). |
| **Детализация** | Достаточно деталей для оценки сроков и стоимости. |
| **Структурированность** | Чёткие заголовки, разделы, списки. |

### 1.2. Ключевые разделы качественного ТЗ для веб-приложения

| № | Раздел | Что описывает |
|---|---|---|
| 1 | **Общие сведения** | Название, цель, бизнес-контекст (суть проекта за 2–3 предложения) |
| 2 | **Функциональные требования** | Что именно должно делать приложение (формы, расчёты, кнопки) |
| 3 | **Требования к интерфейсу** | Как страницы выглядят (без размытых эпитетов: конкретные элементы) |
| 4 | **Технические требования** | Стек технологий, платформы, интеграции, БД |
| 5 | **Требования к данным** | Какие данные хранит БД (структура, поля) |
| 6 | **Нефункциональные требования** | Безопасность, производительность, надёжность |
| 7 | **Состав проекта** | Какие страницы и компоненты будут |
| 8 | **Этапы разработки и сроки** | Последовательность и сроки по этапам |
| 9 | **Критерии приёмки** | Чёткий список: что проверять при сдаче |


## 2. Техническое задание: «BizMath — Веб-приложение для автоматизации экономических расчётов малого бизнеса»

### 2.1. Общие сведения

**Название проекта:** BizMath — веб-приложение для автоматизации экономических расчётов малого бизнеса.

**Краткая суть (Elevator Pitch):**

BizMath — это веб-приложение для малых предприятий и предпринимателей. Пользователь вводит стоимость актива, срок его службы, а также показатели бухгалтерского баланса (оборотные активы, обязательства, запасы) и мгновенно получает правильный расчёт амортизации тремя методами и три коэффициента ликвидности с цветовой индикацией. Все расчёты можно сохранять, редактировать и экспортировать.

**Основная цель:** Обеспечить предпринимателям и бухгалтерам удобный, точный и наглядный инструмент для финансовых расчётов, заменяющий громоздкие Excel-таблицы.

**Область применения:** Внутренний учёт малого бизнеса, образовательный процесс (экономические специальности), помощь ИП и бухгалтерам в работе.

### 2.3. Функциональные требования

**2.3.1. Модуль амортизации**

| ID | Функция | Что должно быть реализовано |
|---|---|---|
| **F-AM-01** | Ввод данных актива | Поля: название (текст), первоначальная стоимость (число), ликвидационная стоимость (число), срок полезного использования (целое число), метод амортизации (выпадающий список: линейный, уменьшаемого остатка, сумма чисел лет) |
| **F-AM-02** | Расчёт линейным методом | Годовая амортизация за все годы по формуле: `(cost — salvage) / years` |
| **F-AM-03** | Расчёт методом уменьшаемого остатка | Годовая амортизация с фактором ускорения 2 и корректировкой остаточной стоимости, остаточная стоимость не опускается ниже ликвидационной |
| **F-AM-04** | Расчёт методом суммы чисел лет | Годовая амортизация по формуле: `(cost — salvage) × remaining_years / sum_of_years` |
| **F-AM-05** | Вывод таблицы | Колонки: Год, Годовая амортизация (руб), Накопленная амортизация (руб), Остаточная стоимость (руб) |
| **F-AM-06** | Визуализация графика | Гистограмма или линейный график годовых отчислений (библиотека Chart.js) |
| **F-AM-07** | Сохранение актива | POST-запрос на сервер, запись в коллекцию `assets` в БД |
| **F-AM-08** | Валидация ввода | Все обязательные поля заполнены, стоимость и срок > 0 |

**2.3.2. Модуль ликвидности**

| ID | Функция | Что должно быть реализовано |
|---|---|---|
| **F-LQ-01** | Ввод данных | Поля: текущие активы (руб), текущие обязательства (руб), запасы (руб), денежные средства (руб), краткосрочные финансовые вложения (руб) |
| **F-LQ-02** | Расчёт коэффициентов | 1. Текущий = текущие активы / текущие обязательства; 2. Быстрый = (активы — запасы) / обязательства; 3. Абсолютный = (деньги + вложения) / обязательства |
| **F-LQ-03** | Цветовая индикация | Текущий: >2 (синий/избыточно), 1.5–2 (зелёный/норма), 1–1.5 (жёлтый/риск), <1 (красный/критично). Аналогично для быстрого (норма 0.7–1) и абсолютного (норма 0.2–0.5) |
| **F-LQ-04** | Вывод даты расчёта | Формат `ДД.ММ.ГГГГ ЧЧ:ММ` через библиотеку dayjs |
| **F-LQ-05** | Сохранение в БД | (Звёздочка) POST-запрос в коллекцию `liquidity_reports` с сохранением всех введённых данных и результатов |

**2.3.3. Модуль управления активами (CRUD)**

| ID | Функция | Что должно быть реализовано |
|---|---|---|
| **F-CR-01** | Просмотр списка | Таблица всех активов из БД. Колонки: ID, Название, Первоначальная стоимость, Срок службы, Метод, Дата создания |
| **F-CR-02** | Редактирование | Кнопка «Редактировать» → форма с предзаполненными данными → PUT-запрос → обновление в БД |
| **F-CR-03** | Удаление | Кнопка «Удалить» → подтверждение → DELETE-запрос |
| **F-CR-04** | Фильтрация по методу | GET-запрос с параметром `?method=linear` и аналогично |

### 2.4. Требования к интерфейсу (UI)

- Все страницы с единой шапкой и сеткой Bootstrap.
- Страница амортизации: форма слева, таблица справа, график под таблицей.
- Страница ликвидности: форма сверху, три карточки коэффициентов под формой, график (звёздочка) под карточками.
- Страница активов: таблица, для каждой строки кнопки «Редактировать» (иконка карандаша) и «Удалить» (иконка корзины).
- Использовать Bootstrap Icons для наглядности.
- Цветовая индикация коэффициентов: зелёный фон для нормы, жёлтый — риск, красный — критично, синий — избыток.

### 2.5. Технические требования

**2.5.1. Технологический стек**

| Компонент | Технология | Почему выбран |
|---|---|---|
| **Фреймворк** | Nuxt.js (Vue 3) | Серверный рендеринг, маршрутизация, удобство разработки |
| **UI** | Bootstrap 5 + Bootstrap Icons | Адаптивность, минимальные стили, хорошая совместимость с Nuxt |
| **Графики** | Chart.js (vue-chartjs) | Лёгкая, популярная, подходит для финансовых графиков |
| **HTTP-клиент** | `$fetch` / Axios | Встроенный + отдельно axios для соответствия критерию |
| **Дата/время** | dayjs | Лёгкая альтернатива moment.js, поддержка форматов |
| **Бэкенд** | Nuxt server routes (встроенный) | Без отдельного сервера, всё в одном проекте, проще деплой |
| **База данных** | MongoDB + Mongoose | Гибкая документоориентированная БД, бесплатно в облаке |
| **Хостинг БД** | MongoDB Atlas | Бесплатный кластер, доступ из любого места |

**2.5.2. Архитектура**

```
Клиент (браузер)
    │
    ├── Nuxt.js (Vue 3) — фронтенд
    │   └── страницы: /, /amortization, /liquidity, /assets-list
    │
    └── Серверные API (Nuxt server routes)
        │
        ├── GET /api/assets → список активов
        ├── POST /api/assets → создать актив
        ├── PUT /api/assets/:id → обновить актив
        ├── DELETE /api/assets/:id → удалить актив
        │
        └── MongoDB (Mongoose)
```

**2.5.3. Интеграции и внешние системы**

- **Chart.js** — генерация графиков на стороне клиента (встроенная, без API).
- **MongoDB Atlas** — облачная БД.
- API-шлюз не требуется.

### 2.7. Нефункциональные требования

**Безопасность:**

- HTTPS используется при деплое (Vercel предоставляет автоматически)
- Валидация всех входных данных на клиенте и на сервере
- JWT-аутентификация: **НЕ требуется** (учебный проект, офлайн-режим допускается)

**Производительность:**

- Страницы загружаются менее чем за 2 секунды в условиях учебного сервера
- Графики отрисовываются без задержек при пересчёте

**Надёжность:**

- При отсутствии подключения к БД расчёты остаются доступными (фронтенд-режим)
- Сохранение активов возможно только при подключении к БД
